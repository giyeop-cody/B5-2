# 아키텍처 및 구현 결정 기록 (Architectural Decision Records - ADR)

본 문서는 `FastAPI 기반 CRUD 웹 서비스 구축` 미션을 수행함에 있어, 실제 구현에서 특정 설계 및 구현 방식을 채택한 **구체적인 이유, 목적, 그리고 이를 통해 얻는 구조적 이점**을 상세히 기록한 문서입니다.

---

## ADR 1: 계층형 아키텍처 (Layered Architecture) 도입

### 1. 배경 및 문제 상황
FastAPI는 매우 직관적이고 유연한 프레임워크여서, 단일 라우터 함수 내에 DB 연결, SQL 쿼리 실행, 비즈니스 로직 검증, HTML 렌더링을 모두 집어넣는 방식으로 개발하기 쉽습니다. 그러나 이 방식은 코드가 길어지고 복잡해질수록 유지보수가 극도로 어려워지며, 특정 비즈니스 로직만 독립적으로 테스트하는 것이 불가능해집니다.

### 2. 결정 사항
애플리케이션의 책임을 `Router`, `Service`, `Repository`, `Model/Schema` 4대 계층으로 완벽히 분리하여 구현합니다.

### 3. 이유 및 목적
- **관심사 분리 (Separation of Concerns):** 각 레이어가 단 하나의 고유한 책임에만 집중하도록 만듭니다. 라우터는 HTTP 요청/응답 처리에만, 서비스는 비즈니스 규칙 처리 및 검증에만, 저장소는 데이터베이스 I/O에만 집중합니다.
- **모듈화 및 확장성 확보:** 추후 DB 저장소가 SQLite에서 PostgreSQL이나 MySQL로 바뀌더라도, 라우터나 서비스 계층의 코드는 단 한 줄도 수정할 필요 없이 저장소(Repository) 코드만 변경하면 되도록 격리(Isolation)하기 위함입니다.

### 4. 구조적 이점
- **테스트 용이성 (Testability):** DB에 연결하지 않고도 Service 계층에 Mock Repository를 주입하여 순수 비즈니스 로직을 완벽히 단위 테스트(Unit Test)할 수 있게 됩니다.
- **가독성 및 유지보수성 향상:** 문제 발생 시 버그의 원인 위치(예: DB 쿼리 오류는 Repository, 유효성 검증 오류는 Service, 경로 렌더링 오류는 Router)를 즉각 파악하고 수정할 수 있습니다.

## ADR 2: Pydantic DTO (Data Transfer Object) 패턴 적용

### 1. 배경 및 문제 상황
SQLAlchemy ORM 모델 객체는 데이터베이스 테이블과 직접 결합되어 있으며, 때로는 내부 세션 상태나 지연 로딩(Lazy Loading) 속성을 갖습니다. 이를 직접 뷰나 클라이언트로 반환하거나 Form 데이터와 매핑하면, 의도치 않은 민감 데이터 노출이나 세션 종료 후 접근 오류(DetachedInstanceError)가 발생할 수 있습니다.

### 2. 결정 사항
`schemas/` 디렉터리에 Pydantic을 활용한 전송 전용 스키마(`MemoCreate`, `MemoUpdate`, `MemoResponse`)를 정의하여 데이터 교환 통로로 사용합니다.

### 3. 이유 및 목적
- **입출력 데이터 규격의 캡슐화:** 클라이언트로부터 입력받는 필드(`title`, `content`)와 서버가 최종 렌더링할 때 필요한 필드(`id`, `created_at` 등 포함)의 스펙 차이를 명확히 구분하기 위함입니다.
- **강력한 타입 및 데이터 검증:** Pydantic의 유효성 검증 기능을 통해 잘못된 타입의 데이터가 Service나 Repository 계층으로 진입하는 것을 원천 차단합니다.

### 4. 구조적 이점
- **데이터 보안 및 무결성 보장:** DB 테이블의 구조 변경이 클라이언트 인터페이스에 직접적인 영향을 주지 않는 완충재(Buffer) 역할을 수행합니다.
- **ORM 디커플링:** `from_attributes=True` (구 `orm_mode`) 옵션을 통해 ORM 객체를 안전한 독립 딕셔너리/객체 구조로 손쉽게 변환하여 템플릿에 전달할 수 있습니다.

## ADR 3: PRG (Post-Redirect-Get) 패턴 및 HTTP 303 상태 코드 채택

### 1. 배경 및 문제 상황
사용자가 HTML Form을 통해 메모를 작성(POST)한 직후 서버가 바로 작성 완료된 HTML 화면을 렌더링해주면, 브라우저의 주소창 상태는 여전히 POST 요청을 가리키고 있습니다. 이 상태에서 사용자가 새로고침(F5)을 누르면 "양식 데이터를 다시 제출하시겠습니까?"라는 경고창이 뜨거나, 동일한 메모가 DB에 중복 생성되는 심각한 UX 및 결함 문제가 발생합니다.

### 2. 결정 사항
등록(`POST /memos/new`), 수정(`POST /memos/{id}/edit`), 삭제(`POST /memos/{id}/delete`) 요청이 성공적으로 완료되면, 반드시 `RedirectResponse(url=..., status_code=303)`을 반환합니다.

### 3. 이유 및 목적
- **새로고침 중복 생성 방지:** 브라우저의 상태를 POST에서 GET으로 완전히 강제 전환시켜, 이후 새로고침을 하더라도 단순히 GET 요청(목록 또는 상세 화면 조회)만 재실행되도록 보장하기 위함입니다.
- **HTTP/1.1 표준 규격 준수:** HTTP 302(Found) 대신 `303 (See Other)`을 명시함으로써, 브라우저가 원본 요청의 메서드(POST)와 상관없이 반드시 `GET` 메서드로 대상 URL을 조회하도록 명확한 브라우저 동작을 유도합니다.

### 4. 구조적 이점
- **사용자 경험(UX) 극대화:** 중복 전송 경고창 없이 매끄럽고 쾌적한 화면 이동을 제공합니다.
- **서버 데이터 무결성 보호:** 비정상적인 중복 Insert/Update/Delete 요청으로 인한 데이터베이스 오염을 원천 차단합니다.

## ADR 4: FastAPI Depends를 활용한 DB 세션 라이프사이클 관리

### 1. 배경 및 문제 상황
웹 서버에서 DB 세션(Connection)을 전역 변수로 공유하거나 수동으로 열고 닫으면, 멀티스레드/비동기 환경에서 트랜잭션 충돌이 발생하거나 세션을 닫지 않아 커넥션 풀 누수(Connection Leak)로 인해 서버가 다운될 수 있습니다.

### 2. 결정 사항
`database.py`에 `get_db()` 제너레이터 함수를 생성하고, `yield` 메커니즘과 FastAPI의 `Depends(get_db)`를 활용하여 각 라우터 함수에 DB 세션을 주입합니다.

### 3. 이유 및 목적
- **요청 단위 트랜잭션 (Per-Request Connection):** 단일 HTTP 요청 주기에 정확히 1:1로 매핑되는 독립적인 DB 세션을 할당하여 트랜잭션 격리성을 유지하기 위함입니다.
- **자동 정리 및 예외 안전성 (Exception Safety):** 실행 도중 비즈니스 예외나 서버 오류가 발생하더라도 `finally` 블록을 통해 `db.close()`가 100% 실행되도록 보장합니다.

### 4. 구조적 이점
- **시스템 안정성 극대화:** 커넥션 풀 고갈을 예방하여 장시간 가동 시에도 서버 안정성을 획득합니다.
- **결합도 감소:** 라우터나 서비스 코드가 세션 생성 및 해제에 대한 관리 책임을 갖지 않고, 주입받은 세션을 사용하는 데만 집중할 수 있습니다.

## ADR 5: 보너스 과제(검색 및 검증)의 계층별 최적 분할 배치

### 1. 배경 및 문제 상황
보너스 과제인 '제목 검색 기능'과 '빈 값 유효성 검증'을 어디에 구현할 것인가에 대한 아키텍처적 결정이 필요합니다. 라우터에서 모두 처리하면 계층 분리 원칙이 깨집니다.

### 2. 결정 사항
- **빈 값 검증 (Validation):** 비즈니스 로직 계층인 `MemoService`에서 수행하며, 오류 발생 시 사용자 안내 메시지를 반환하여 라우터가 폼 화면을 재렌더링하도록 처리합니다.
- **검색 기능 (Search):** 데이터베이스 접근 전담 계층인 `MemoRepository`의 `get_all(db, search)` 메소드에 SQLAlchemy `like` 필터 쿼리로 구현합니다.

### 3. 이유 및 목적
- **비즈니스 규칙과 데이터 추출 규칙의 분리:** '필수값이 비어 있으면 저장되지 않는다'는 비즈니스 정책이므로 Service의 책임이며, '특정 키워드가 포함된 제목을 찾는다'는 데이터 쿼리 최적화 정책이므로 Repository의 책임이기 때문입니다.

### 4. 구조적 이점
- **명확한 설계 원칙 실증:** 확장 기능이 추가되더라도 기존의 계층형 아키텍처 원칙을 견고하게 유지하면서 확장이 가능함(OCP - 개방 폐쇄 원칙)을 증명합니다.
